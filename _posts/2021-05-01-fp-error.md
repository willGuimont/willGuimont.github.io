Error handling is hard. Everything can go wrong anytime. To make reliable software, error handling is central

Uni -> Java, so we got experience with the "mainstream" way of handling them

# A small example

Example what code do?:

{% highlight java %}
User createUser(Request request) {
    String body = request.getBody();
    Map<String, String> json = Json.parse(body);

    String name = json.get("name");
    String jobTitle = json.get("jobTitle");

    return new User(name, jobTitle);
}
{% endhighlight %}

Simply loading json from an HTTP request and parsing it into an object.
Well yes but actually no... That is true for the happy path, but how can that code break?

{% highlight java %}
User createUser(Request request) {
    // If the request has no body, returns null
    String body = request.getBody();
    // If the body is null, then this will throw a NullPointerException (NPE)
    // If the body cannot be parsed as JSON, throws a JsonParsingException
    Map<String, String> json = Json.parse(body);

    // If the parsed JSON does not contain the requested field, will return null
    String name = json.get("name"); 
    String jobTitle = json.get("jobTitle");

    // Inside the constructor, there's a StringUtils.capitalize(name), will throw a NPE
    // Will throw an exception if name or jobTitle is empty
    return new User(name, jobTitle);
}
{% endhighlight %}

hmmm... ok, let's see how to make that code reliable
{% highlight java %}
User createUser(Request request) {
    String body = request.getBody();
    if (body != null) {
        try {
            Map<String, String> json = Json.parse(body);

            String name = json.get("name");
            String jobTitle = json.get("jobTitle");

            if (name != null) {
                String jobTitle = json.get("jobTitle");
                if (jobTitle == null) {
                    try {
                        return new User(name, jobTitle);
                    } catch (EmptyNameException | EmptyJobTitle e) {
                        return null;
                    }
                } else {
                    return null;
                }
            } else {
                return null;
            }
        } catch (JsonParsingException e) {
            return null;
        } catch (KeyNotFoundException e) {
            return null;
        }
    }
}
{% endhighlight %}

oof, that's a lot of code. At least we're reliable! right?

Well actually no, the line

{% highlight java %}
if (jobTitle == null) {
{% endhighlight %}

Should in fact be 

{% highlight java %}
if (jobTitle != null) {
{% endhighlight %}

The code is so complex, it's easy to make mistakes like that.

And here, this is a simple case, we only have two possible fields that could be null.

With all that error handling, it's hard to see what the method actually does...

And if you check the types of the variables, it's really hard to know what could return null. We basically have to know what method could return null and which won't. Especially hard since the standard library does that at multiple places, notably for Map.

The problem is that any variable could be null...

If we don't check nulls, then we are open to NullPointerExceptions. So we're best to avoid nulls, except that we can't always, like when we use the standard java library.

Also, when using other classes, it is hard to know what are the possible exception that could be thrown. Checked exception kind of helps with that, if not that they seem frowned upon. Since when you add a checked exception, you often have to change de signature of all method calling that method up until the error is catched.

As Robert C. Martin said in *Clean Code*[^Martin]:

> Checked exceptions can sometimes be useful if you are writing a critical library: You must catch them. But in general application development the dependency costs outweigh the benefits.
>
> -- Robert C. Martin[^Martin]

So maybe exceptions aren't the *best* way to handle error... If the cost of maintaining is so great that you end up doing weird things.

So we often end up using RuntimeExceptions that don't need to explicitely be declared in the signature. But that comes with a cost, callers of those methods, need to know what could go wrong and handle the exceptions accordingly

Also, Exceptions are kinda like Gotos, they make the program flow jumps around and makes the code often harder to understand.

So, it seems that to write reliable code, we must be paranoiac and know all of the documentation of every library we use...

Surely there could be another way of handling errors...

# Another way

For an university project, we had *carte blanche* for which technologies to use.

Know that Go was getting pretty popular, we considered taking that language. Before actually committing to using that language, we did a simple test. We wrote a TODO app backend. The interesting thing with that project is that it allowed my team and I to discover a big pain point with Go. The error handling.

First let's explain a bit how Go handle errors

Go uses an explicit error return. Here's an example from the official website[^Go]

{% highlight go %}
f, err := os.Open("filename.ext")
if err != nil {
    log.Fatal(err)
}
// do something with the open *File f
{% endhighlight %}

Here the `Open` function returns both what we want, the file, and a `err` containing an error if any, or `nil` otherwise.

The programmer have to check wether `err` is `nil` or not before actually using the returned file.

So, in our TODO app, we tried to handle error correctly, but we quickly ended in a nested mess of `if err != null`. That way of handling error puts a lot of work onto the programmer to make the error plumbing.

So we ended up not choosing Go, instead we went with Scala. A more... functional choice
